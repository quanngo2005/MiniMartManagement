# JWT Cookie Authentication

This document explains how the MiniMart authentication system works and how other developers should use it from frontend, backend, and database setup code.

## Overview

MiniMart auth is integrated into the existing `backend/MiniMart` project. It does not use a separate `ApplicationUser` model. The account source is the existing `Employee` table, and authorization roles come from the existing `Role` table.

The system uses two tokens:

- `access_token`: a JWT stored in an HttpOnly Secure cookie. It is short lived and used to authenticate API requests.
- `refresh_token`: a random secure token stored in an HttpOnly Secure cookie. It is long lived, stored in the database only as a hash, and used to issue a new access token.

The browser does not store tokens in `localStorage` or `sessionStorage`.

## Main Files

- `Controllers/AuthController.cs`: exposes `/api/auth/*` endpoints and sets/deletes auth cookies.
- `Services/AuthService.cs`: login, refresh rotation, logout, lockout, device limit, password hashing.
- `Services/JwtService.cs`: creates JWT access tokens, creates refresh tokens, hashes refresh tokens.
- `Models/RefreshToken.cs`: database entity for refresh token sessions.
- `Middleware/CsrfMiddleware.cs`: protects unsafe cookie-authenticated requests.
- `Middleware/ExceptionMiddleware.cs`: returns consistent JSON error responses.
- `Extensions/ServiceExtensions.cs`: registers JWT auth, DI services, and authorization policies.
- `Settings/JwtSettings.cs`: strongly typed JWT settings.
- `Migrations/20260624170027_AddCookieAuthentication.cs`: adds auth columns, refresh token table, and role seed SQL.

## Configuration

Configure JWT settings in `backend/MiniMart/appsettings.json`:

```json
"Jwt": {
  "SecretKey": "CHANGE_ME_TO_A_REAL_32_BYTE_SECRET_KEY",
  "Issuer": "MiniMart",
  "Audience": "MiniMart",
  "AccessTokenExpirationMinutes": 15,
  "RefreshTokenExpirationDays": 7,
  "RememberMeRefreshTokenExpirationDays": 30
}
```

Production requirements:

- Replace `Jwt:SecretKey` with a real secret of at least 32 characters.
- Use HTTPS, because auth cookies are marked `Secure`.
- Existing employees must have `PasswordHash` values in the PBKDF2 format generated by `AuthService`.

Run the migration before using auth:

```bash
dotnet ef database update --project backend/MiniMart --startup-project backend/MiniMart
```

## Cookies

Login and refresh set two cookies:

| Cookie | Purpose | HttpOnly | Secure | SameSite | Path | Lifetime |
| --- | --- | --- | --- | --- | --- | --- |
| `access_token` | Authenticates normal API requests | Yes | Yes | Strict | `/` | 15 minutes |
| `refresh_token` | Issues a new token pair | Yes | Yes | Strict | `/api/auth` | 7 days or 30 days with remember me |

Because both tokens are HttpOnly, frontend JavaScript cannot read them. The browser sends them automatically when calling the backend.

## CSRF Requirement

Cookie auth needs CSRF protection because browsers automatically attach cookies. MiniMart uses a double-submit token.

Frontend must first call:

```http
GET /api/auth/csrf-token
```

The backend returns a readable `XSRF-TOKEN` cookie and also returns the token in the response body. For every unsafe method, frontend must send the same value in this header:

```http
X-XSRF-TOKEN: <value from XSRF-TOKEN cookie>
```

Unsafe methods are:

- `POST`
- `PUT`
- `PATCH`
- `DELETE`

Safe methods do not require the CSRF header:

- `GET`
- `HEAD`
- `OPTIONS`
- `TRACE`

## Login Flow

1. Frontend calls `GET /api/auth/csrf-token`.
2. Frontend calls `POST /api/auth/login` with `X-XSRF-TOKEN`.
3. `AuthService.LoginAsync` loads `Employee` with `Role`.
4. The service verifies:
   - username exists
   - password matches PBKDF2 hash
   - employee status is `Active`
   - role is active
   - account is not locked
5. Failed logins increment `FailedLoginAttempts`.
6. After 5 failed attempts, `LockoutEnd` is set for 15 minutes.
7. Successful login resets lockout fields.
8. Server creates an access JWT and a refresh token.
9. Server stores only the refresh token hash in `RefreshTokens`.
10. Server sets `access_token` and `refresh_token` cookies.

Example request:

```http
POST /api/auth/login
Content-Type: application/json
X-XSRF-TOKEN: <csrf-token>

{
  "username": "admin.test",
  "password": "Admin@123",
  "rememberMe": true
}
```

The response body contains user data and access expiry. It does not return the raw tokens.

Development/test login accounts:

| Role | Username | Password |
| --- | --- | --- |
| Admin | `admin.test` | `Admin@123` |
| Manager | `manager.test` | `Manager@123` |

## Authenticated Requests

For protected endpoints, frontend should call the backend with credentials enabled. The browser sends `access_token` automatically.

JavaScript example:

```js
await fetch("/api/auth/me", {
  method: "GET",
  credentials: "include"
});
```

Backend validation works like this:

1. ASP.NET JWT Bearer checks for an Authorization header.
2. If no bearer token exists, `ServiceExtensions` reads `access_token` from cookies.
3. JWT Bearer validates issuer, audience, expiry, and signature.
4. Role policies use the role claim from the JWT.

## Refresh Flow

When the access token expires, frontend calls:

```http
POST /api/auth/refresh-token
X-XSRF-TOKEN: <csrf-token>
```

The backend reads `refresh_token` from the cookie.

Refresh behavior:

- The raw refresh token is hashed.
- The hash is looked up in `RefreshTokens`.
- If the token is active and not expired, the old token is revoked.
- A new access token and refresh token are created.
- The old row stores `ReplacedByTokenHash`.
- A new refresh token row is inserted.
- Cookies are replaced with the new token pair.

This is refresh token rotation. Each refresh token is single use.

## Refresh Token Reuse Detection

If an already revoked refresh token is used again, MiniMart treats it as possible token theft.

When reuse is detected:

- `AuthService.RefreshTokenAsync` revokes the whole token family.
- The request fails with unauthorized.
- The user must log in again.

This limits damage if a stolen refresh token is replayed after the real user has already refreshed.

## Logout

Current device logout:

```http
POST /api/auth/logout
X-XSRF-TOKEN: <csrf-token>
```

Behavior:

- Revokes the current refresh token if present.
- Deletes `access_token`.
- Deletes `refresh_token`.

All devices logout:

```http
POST /api/auth/logout-all
X-XSRF-TOKEN: <csrf-token>
```

Behavior:

- Revokes all active refresh tokens for the current employee.
- Deletes current cookies.

## Device Limit

Each employee can have up to 3 active refresh token sessions.

After a successful login, `AuthService` checks active tokens for the employee. If there are more than 3, older sessions are revoked.

## Roles And Policies

Roles are stored in the `Roles` table and referenced by `Employee.RoleId`.

The auth migration seeds these role names if they do not exist:

- `Admin`
- `Manager`
- `Cashier`
- `Warehouse`
- `Staff`

Policies:

| Policy | Allowed roles |
| --- | --- |
| `ManagerUp` | `Admin`, `Manager` |
| `WarehouseUp` | `Admin`, `Manager`, `Warehouse` |
| `AnyEmployee` | `Admin`, `Manager`, `Cashier`, `Warehouse`, `Staff` |

Controller usage example:

```csharp
[Authorize(Policy = "ManagerUp")]
```

## Endpoint Summary

| Endpoint | Auth | Purpose |
| --- | --- | --- |
| `GET /api/auth/csrf-token` | No | Get CSRF token for unsafe requests |
| `POST /api/auth/login` | No, but CSRF required | Login and set token cookies |
| `POST /api/auth/refresh-token` | Refresh cookie | Rotate refresh token and set new cookies |
| `POST /api/auth/logout` | `AnyEmployee` | Logout current device |
| `POST /api/auth/logout-all` | `AnyEmployee` | Logout all devices |
| `POST /api/auth/register` | `ManagerUp` | Create employee account |
| `POST /api/auth/change-password` | `AnyEmployee` | Change password and revoke sessions |
| `GET /api/auth/me` | `AnyEmployee` | Get current user |
| `POST /api/auth/toggle-active/{employeeId}` | `ManagerUp` | Activate or deactivate an employee |

## Middleware Order

`Program.cs` wires middleware in this order:

1. `ExceptionMiddleware`
2. Swagger or production error/HSTS
3. HTTPS/static/routing
4. `CsrfMiddleware`
5. `UseAuthentication`
6. `UseAuthorization`
7. Controllers

Why this order matters:

- `ExceptionMiddleware` runs early so it can convert auth/domain errors into JSON.
- `CsrfMiddleware` runs before authentication logic handles unsafe requests.
- `UseAuthentication` must run before `UseAuthorization`, otherwise policies cannot see the authenticated user.

## Frontend Checklist

- Call `GET /api/auth/csrf-token` when the app starts or before the first unsafe request.
- Send `credentials: "include"` on fetch/Axios requests.
- Send `X-XSRF-TOKEN` on `POST`, `PUT`, `PATCH`, and `DELETE`.
- Do not store access or refresh tokens in frontend storage.
- On `401`, call `POST /api/auth/refresh-token`, then retry the original request once.
- On refresh failure, send user back to login.

## Backend Checklist

- Use `[Authorize(Policy = "...")]` on protected controllers/actions.
- Use role names exactly as seeded: `Admin`, `Manager`, `Cashier`, `Warehouse`, `Staff`.
- Do not expose `PasswordHash` in request DTOs or API responses.
- Create employees through auth/register or hash passwords with the same PBKDF2 format.
- Keep `Jwt:SecretKey` out of source-controlled production config.

## Common Failure Cases

| Symptom | Likely cause |
| --- | --- |
| Login returns invalid CSRF token | Missing `X-XSRF-TOKEN` header |
| Protected request returns 401 | Missing/expired `access_token` cookie |
| Refresh returns 401 | Missing, expired, revoked, or reused refresh token |
| Login returns account inactive | `Employee.Status` is not `Active` or `Role.Status` is false |
| Login keeps failing for old employees | Stored `PasswordHash` is not in the PBKDF2 format expected by `AuthService` |
| Cookies not saved in browser | HTTPS missing, cross-site cookie settings mismatch, or credentials not included |
