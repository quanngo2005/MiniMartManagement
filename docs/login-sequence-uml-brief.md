# UML-ready codebase brief — login sequence

## Scope and evidence

- Repository / module: `MiniMartManagement`, limited to `frontend/mini_mart_management_mobile_app` and `backend/MiniMart`.
- Runtime and architecture: Flutter client using Provider → Repository → HTTP Service; ASP.NET Core Web API using Controller → Service → Repository → EF Core/SQL Server.
- System boundaries: `FlutterClient` (mobile app) and `MiniMartApi` (ASP.NET Core API); `MiniMartDatabase` is the API's SQL Server persistence boundary.
- Scope used: interactive username/password login, including the required CSRF-token preflight, cookie issuance, client-side state update, and role-based navigation.
- Sequence diagrams: included scenario: `Interactive employee login`.
- Entry points inspected: `FlutterClient::LoginScreen._handleLogin`, `FlutterClient::AuthProvider.login`, `FlutterClient::AuthService.login`, `MiniMartApi::AuthController.GetCsrfToken`, `MiniMartApi::AuthController.Login`, and `MiniMartApi::AuthService.LoginAsync`.
- Important exclusions: refresh-token rotation, logout, registration, and authenticated requests after login are outside this brief. A class diagram is intentionally not included because only a sequence diagram was requested.

## System introduction

The Flutter application starts at `LoginScreen` and registers an `AuthProvider` backed by `AuthRepository` and `AuthService`. On submission, the screen delegates login to the provider; after a successful result it replaces the login route with a screen selected from the returned employee role. [frontend/mini_mart_management_mobile_app/lib/app.dart:MiniMartManagementApp.build](../frontend/mini_mart_management_mobile_app/lib/app.dart#L62), [frontend/mini_mart_management_mobile_app/lib/screens/login_screen.dart:_LoginScreenState._handleLogin](../frontend/mini_mart_management_mobile_app/lib/screens/login_screen.dart#L39)

Before posting credentials, the client requests a CSRF token. The API returns the token in the standard `ApiResponse` envelope and writes it to the `XSRF-TOKEN` cookie. The login POST carries the token in both the `X-XSRF-TOKEN` header and cookie; `CsrfMiddleware` rejects unsafe requests whose values are missing or do not match. [frontend/mini_mart_management_mobile_app/lib/services/auth_service.dart:AuthService._fetchCsrfToken](../frontend/mini_mart_management_mobile_app/lib/services/auth_service.dart#L71), [backend/MiniMart/Controllers/AuthController.cs:AuthController.GetCsrfToken](../backend/MiniMart/Controllers/AuthController.cs#L26), [backend/MiniMart/Middleware/CsrfMiddleware.cs:CsrfMiddleware.InvokeAsync](../backend/MiniMart/Middleware/CsrfMiddleware.cs#L28)

The API looks up the employee with its role, verifies the PBKDF2 password hash, validates active/unlocked account state, generates access and refresh tokens, and stores only the refresh-token hash. The controller writes both plaintext tokens as secure, HTTP-only cookies and returns an `AuthResponse` containing the user profile and access-token expiry. [backend/MiniMart/Services/Implementations/AuthService.cs:AuthService.LoginAsync](../backend/MiniMart/Services/Implementations/AuthService.cs#L40), [backend/MiniMart/Services/Implementations/JwtService.cs:JwtService](../backend/MiniMart/Services/Implementations/JwtService.cs#L13), [backend/MiniMart/Controllers/AuthController.cs:AuthController.SetTokenCookies](../backend/MiniMart/Controllers/AuthController.cs#L119)

## Sequence diagram briefs

### Scenario: Interactive employee login

- Trigger and expected outcome: An employee submits credentials in `FlutterClient::LoginScreen`; on success the client retains the returned `EmployeeUser` in `AuthProvider`, its cookie client retains the issued cookies, and the UI moves to the role-specific destination.
- Participants (left to right): Employee, `FlutterClient::LoginScreen`, `FlutterClient::AuthProvider`, `FlutterClient::AuthRepository`, `FlutterClient::AuthService` / cookie HTTP client, `MiniMartApi::CsrfMiddleware`, `MiniMartApi::AuthController`, `MiniMartApi::AuthService`, `MiniMartApi::EmployeeRepository`, `MiniMartApi::JwtService`, `MiniMartApi::RefreshTokenRepository`, `MiniMartDatabase`.
- System boundary grouping: `FlutterClient` contains the UI, provider, repository, service, and cookie client. `MiniMartApi` contains middleware, controller, service, repositories, and JWT service. `MiniMartDatabase` is reached only through the API repositories.

#### Diagram construction notes

- Async messages/events: repository, HTTP, service, and EF Core interactions are asynchronous in source (`Future`/`Task`); the user tap is a UI event. A standard UML sequence diagram may show these as synchronous call arrows with an `async` label to keep the diagram readable.
- Authorization, validation, and transaction boundaries: login itself has no `[Authorize]` attribute. CSRF validation is enforced globally for the POST. Account/credential checks are in `AuthService`. No explicit database transaction boundary is visible; employee failure-state saving, refresh-token creation, and device-limit revocations use separate repository saves, so end-to-end atomicity is not verified.
- Same-named types across system boundaries: both runtimes define `AuthService`. Keep them distinct as `FlutterClient::AuthService` (HTTP client and response parsing) and `MiniMartApi::AuthService` (credential validation and token orchestration); they are not the same type and have no ownership relationship across HTTP.
- Inferences or unresolved details: Browser/Flutter cookie persistence is inferred from `CookieClient` parsing `Set-Cookie`; exact production cookie behavior depends on the platform HTTP implementation. The exception-to-status mapping must be read from `ExceptionMiddleware` before drawing precise 401/403 paths.

### Sequence diagram — Interactive employee login

Create a Sequence Diagram for interactive employee login across three system boundaries: FlutterClient, MiniMartApi, and MiniMartDatabase. An Employee submits credentials to FlutterClient::LoginScreen, which calls FlutterClient::AuthProvider, AuthRepository, and AuthService. FlutterClient::AuthService first sends GET /api/auth/csrf-token to MiniMartApi::AuthController; the API returns csrfToken and sets XSRF-TOKEN.
