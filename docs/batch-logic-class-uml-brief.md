## System introduction

`Batch` represents an inventory lot: it carries product identity, manufacture/expiry dates, import cost, imported and remaining quantities, availability, provenance, and an EF Core concurrency token. In source, `BatchProvenance` is an enumeration with literals `Receipt` and `StockCountAdjustment`. The target diagram generator accepts only class and interface elements, so the generated class diagram must keep those literals as the constrained `Batch.Provenance: Receipt | StockCountAdjustment` attribute rather than create a standalone enum node. Its `QuantityRemaining` is explicitly the source of truth for expiry-aware allocation; `Product.StockQuantity` is a fast-read cache. [Batch.cs:7](../backend/MiniMart/Models/Batch.cs#L7), [BatchProvenance.cs:3](../backend/MiniMart/Models/Enums/BatchProvenance.cs#L3), [Product.cs:17](../backend/MiniMart/Models/Product.cs#L17)

The standalone `api/batches` / `odata/Batches` endpoint exposes CRUD through `BatchesController`, guarded by `AnyEmployee` for reads and `ManagerUp` for mutations. The controller depends on `IBatchService`; `BatchService` validates referenced product and receipt IDs through `IBatchRepository`, maps DTOs with AutoMapper, and delegates persistence to `BatchRepository`. [BatchesController.cs:10](../backend/MiniMart/Controllers/BatchesController.cs#L10), [BatchService.cs:11](../backend/MiniMart/Services/Implementations/BatchService.cs#L11), [BatchRepository.cs:8](../backend/MiniMart/Repositories/Implementations/BatchRepository.cs#L8)

A batch has one required `Product` and an optional `Receipt`; each product and receipt exposes a collection of batches. EF Core configures both foreign keys with restrictive delete behavior. Batches also have zero or more `InventoryTransaction` records; the transaction-side batch foreign key is optional and also restrictive. These are persistent associations, not ownership/composition relationships. [MiniMartDbContext.cs:101](../backend/MiniMart/Data/MiniMartDbContext.cs#L101), [MiniMartDbContext.cs:210](../backend/MiniMart/Data/MiniMartDbContext.cs#L210)

Receipt creation can construct batches directly from `ReceiptBatchLineDto` values, initially with zero remaining quantity and inactive status. Completing a pending receipt then raises both the cached product stock and each batch's remaining quantity, creates an import `InventoryTransaction`, and marks the receipt complete. [ReceiptService.cs:48](../backend/MiniMart/Services/Implementations/ReceiptService.cs#L48), [ReceiptService.cs:136](../backend/MiniMart/Services/Implementations/ReceiptService.cs#L136)

Stock-count review also modifies the model: negative variance deducts from eligible batches ordered by expiry date, receipt import date, and batch ID; positive variance creates a batch with `StockCountAdjustment` provenance. The stock-count repository adds created batches and inventory transactions in its verified transaction callback. [StockCountService.cs:269](../backend/MiniMart/Services/Implementations/StockCountService.cs#L269), [StockCountService.cs:321](../backend/MiniMart/Services/Implementations/StockCountService.cs#L321), [IStockCountRepository.cs:16](../backend/MiniMart/Repositories/Interfaces/IStockCountRepository.cs#L16)

### Diagram construction notes

- Suggested diagram boundary: one `MiniMartApi` class diagram with four packages: API/CRUD, batch domain, receipt/inventory workflow, and stock-count workflow. Draw `MiniMartDatabase` only as a persistence note behind `MiniMartDbContext`, not as a domain class.
- Types to omit as framework/plumbing: `ControllerBase`, EF Core `DbSet`/LINQ types, OData framework types, AutoMapper's concrete runtime types, and generic `ApiResponse<T>`.
- Do not draw `Receipt` *-- `Batch` composition. Although receipts create a collection of batches, the entity permits `ReceiptId == null`, and the EF relation is configured as restrictive rather than cascading. [Batch.cs:40](../backend/MiniMart/Models/Batch.cs#L40), [MiniMartDbContext.cs:101](../backend/MiniMart/Data/MiniMartDbContext.cs#L101)
- Do not model `Product.StockQuantity` as an independently authoritative aggregate value: source comments identify active `Batch.QuantityRemaining` as its source of truth. [Product.cs:17](../backend/MiniMart/Models/Product.cs#L17)
- Because the target diagram generator supports only class/interface elements, do not create a `BatchProvenance` diagram element. Show `Provenance: Receipt | StockCountAdjustment` directly in the `Batch` attribute compartment, with no enum relationship edge. The source enum remains documented in the inventory for traceability. [Batch.cs:32](../backend/MiniMart/Models/Batch.cs#L32), [BatchProvenance.cs:3](../backend/MiniMart/Models/Enums/BatchProvenance.cs#L3)
- Inference/unresolved relationship: `Batch.RowVersion` and `Product.RowVersion` are EF Core concurrency tokens, but no batch-specific conflict-handling branch was inspected; show them as `byte[]` attributes without inventing conflict behavior. [MiniMartDbContext.cs:118](../backend/MiniMart/Data/MiniMartDbContext.cs#L118)
